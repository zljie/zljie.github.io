> 作者：Johnson Zhao
> 日期：2026-05-28  
> 标签：Enterprise Architecture / OSI Semantic Model / EA拆解 / Semantic Ontology / Skill Pipeline  

---

## 前言：两条 Skill，串起从业务到本体的完整链路

把一个业务领域变成可被 AI 理解、可执行的语义本体，通常面临两个关键问题：

**问题一**：业务侧的复杂度太高——流程、数据、系统、角色搅在一起，很难一次性说清楚"要建什么本体"。  
**问题二**：建模侧没有标准约束——不同人建的本体粒度不同、关系描述各异，难以横向对齐和复用。

本文介绍一条我从实践中总结出的 pipeline，用两条 Skill 解决这两个问题：

- **EA 业务拆解**（`ea-business-decomposition-cskill`）：用第一性原理 × MECE × EA 四层，把业务域从"目的与约束"逐层拆解到"能力、流程、数据、系统、集成事件"，输出结构化的《分析结果》+ 《逻辑推导》。

- **OSI 模型生成**（`osi-model-generator-skill`）：依据 OSI Core 规范，把 EA 拆解的产出转成符合 `semantic_model` YAML 的语义本体，可校验、可导入、可视化，输出完整的 datasets / relationships / metrics / actions / rules。

**这两条 Skill 串起来，就是"业务 → 本体"的端到端闭环：**

```
业务领域（输入）
    ↓
EA 业务拆解
    ├─ 《分析结果》  →  能力地图、流程变体、系统边界、事件流
    └─ 《逻辑推导》  →  Root Drivers → KPI → MECE 校验
    ↓
OSI 模型生成
    ├─ datasets（对象粒度 + 主键 + 字段）
    ├─ relationships（可验证的 join 关系）
    ├─ metrics（口径显式化的指标）
    ├─ actions + rules（行为层 + 治理门禁）
    └─ validate.py 校验通过
```

## 一、EA 业务拆解：把业务域从混沌拆成结构

### 1.1 为什么要先做 EA 拆解

在直接建模之前，业务域往往是一团"需求说话"——"我们要做一个供应商管理的模块"、"销售订单要能跨系统查"。

EA 拆解的第一步是**把需求翻译成结构**：

- 这个领域的**本质**是什么？
- 哪些**根本诉求**不可再分？
- 支撑这些诉求的**能力**有哪些？
- 这些能力在哪些**系统**里执行？
- 系统之间靠什么**事件**串联？

经过这层拆解，"供应商管理模块"会变成：供应商准入 → 供应商分级 → 采购协同 → 绩效评估，一套有边界、有关系、有数据载体的能力体系。

### 1.2 EA 拆解的四个步骤

以「需求到采购（DTP）」为例，说明四步拆解的过程：

**Step 1：第一性原理——找 Root Drivers**

> 本质 = 在正确的时间/成本/风险边界内，获得/交付正确的资源（物料/服务/信息/资金），保障业务连续性与可持续增长。

由此推导出 3 个根本诉求（Root Drivers）：

| Root Driver | 不可再分的原因 |
|------------|--------------|
| **RD-1 供应可见** | 不管谁提需求，首先要知道"有没有库存/谁能供"，这是采购决策的前提 |
| **RD-2 合规可控** | 采购金额超阈值谁批、供应商资质是否合规，这些控制不能事后补 |
| **RD-3 履约可追** | 订单下了能不能到、什么时候到、收没收到货，收不到怎么办 |

**Step 2：MECE 拆解——能力域 + 流程族**

从 Root Drivers 推导 KPI（可衡量、可追溯），再映射到能力域：

```
DTP 能力地图

├── 需求感知
│   ├── 需求捕获（计划/手工/AI 推荐）
│   └── 需求验证（库存查重/预算检查）
│
├── 寻源执行
│   ├── 供应商寻源与准入
│   ├── 比价与定价
│   └── 合同签订
│
├── 采购执行
│   ├── 请购 / 审批
│   ├── 订单下达
│   └── 交期跟踪
│
├── 履约确认
│   ├── 收货与验收
│   ├── 入库与质检
│   └── 不符与争议处理
│
└── 绩效治理
    ├── 供应商绩效考核
    ├── 采购成本分析
    └── 审计与追溯
```

流程变体用维度组合穷举：需求来源（计划/手工）× 利库（是/否）× 采购方式（集采/分散）× 收货模式（实物/服务/免检），形成 4-8 个必须覆盖的流程变体。

**Step 3：EA 四层映射**

| 层级 | 产出 |
|------|------|
| **业务架构（BA）** | 能力地图 + 流程主干 + 控制点（合规/预算/审批） |
| **数据架构（DA）** | 主数据（供应商/物料/组织）+ 交易数据（PR/PO/GR）+ 指标口径 |
| **应用架构（AA）** | 系统清单（ERP/SRM/WMS/MDM）与主责边界 |
| **技术架构（TA）** | 集成方式（API/消息）+ Event Catalog（`PRCreated` / `POReleased` / `GRPosted`）|

**Step 4：两份交付物**

- **《分析结果》**：执行摘要 + OKR/KPI + 能力地图 + 流程变体 + 系统清单 + 数据资产 + 治理建议
- **《逻辑推导》**：推导链路图 + MECE 校验 + 假设与不确定性

### 1.3 EA 拆解的输出如何进入 OSI 建模

EA 拆解的产出天然对应 OSI 模型的核心构件：

| EA 拆解产出 | OSI 模型落点 |
|------------|-------------|
| 能力域中的核心业务对象（供应商/物料/订单） | `datasets` |
| 业务对象之间的关系（PR → PO → GR → IV） | `relationships` |
| KPI 与指标口径 | `metrics` |
| 流程中的关键动作（审批/下达/验收） | `behavior.actions` |
| 控制点（合规/预算/资质检查） | `behavior.rules` |

这意味着：**EA 拆解不是终点，而是 OSI 建模的输入规范**。两者的衔接让"业务说清楚"和"本体建标准"不再需要人工来回对齐。

## 二、OSI 模型生成：把结构翻译成可执行的语义本体

### 2.1 为什么需要 OSI 标准

EA 拆解给出了"业务上需要什么"，OSI 模型给出了"机器能读懂什么"。

OSI（Open Semantic Interchange）是一个 vendor-neutral 的语义模型交换标准，用 YAML 描述业务对象的结构、关系、指标和行为。它的核心价值在于：**让不同 BI/AI 工具共享同一语义定义**，消除 Metric Drift（不同平台 KPI 不一致）和 LLM 的语义幻觉。

OSI v0.1.2 引入了 `behavior` 行为层（actions / rules / effects），从"只读查询"向"可执行闭环"延伸——这与企业级语义本体的方向完全吻合。

### 2.2 OSI 模型生成的四层结构

OSI 语义模型由四层构成，层与层之间有明确的依赖关系：

**第一层：datasets（对象粒度）**

每个 dataset 代表一个业务对象/事实粒度，必须包含：

- `name`：逻辑名（snake_case，面向业务）
- `source`：物理表名（`db.schema.table`）
- `primary_key`：唯一键，用于跨系统身份对齐
- `fields`：可用于分组/过滤/聚合的字段，每个字段必须声明 `expression.dialects[]`

```yaml
- name: suppliers
  source: sap.p2p.suppliers
  primary_key: [supplier_id]
  fields:
    - name: supplier_id
      expression:
        dialects:
          - dialect: ANSI_SQL
            expression: suppliers.supplier_id
    - name: status
      expression:
        dialects:
          - dialect: ANSI_SQL
            expression: suppliers.status
      description: 供应商状态（Active / Blocked / Inactive）
```

**第二层：relationships（可验证的关系）**

每条 relationship 必须满足可 join 的引用完整性约束：

```yaml
relationships:
  - name: pr_items_to_pr_header
    from: purchase_requisition_items
    to: purchase_requisitions
    from_columns: [purchase_requisition_id]
    to_columns: [purchase_requisition_id]
  - name: pr_to_suppliers
    from: purchase_requisitions
    to: suppliers
    from_columns: [supplier_id]
    to_columns: [supplier_id]
```

**第三层：metrics（口径显式化）**

每个 metric 必须包含计算口径的描述，避免歧义：

```yaml
metrics:
  - name: pr_count
    expression:
      dialects:
        - dialect: ANSI_SQL
          expression: COUNT(DISTINCT purchase_requisitions.purchase_requisition_id)
    description: 请购单数量（按去重 PR ID 计数）
  - name: on_time_delivery_rate
    expression:
      dialects:
        - dialect: ANSI_SQL
          expression: SUM(CASE WHEN goods_receipts.received_date <= goods_receipts.scheduled_date THEN 1 ELSE 0 END) / COUNT(*)
    description: 按时交货率 = 按时交货数 / 总交货数
```

**第四层：behavior（可执行闭环）**

behavior 层是 OSI v0.1.2 的核心扩展，包含 actions（动作）和 rules（规则）：

```yaml
behavior:
  namespace: "DTP"
  behavior_layer_version: "0.1"
  actions:
    - id: suppliers/block
      title: 冻结/阻断供应商
      kind: command
      operation: block
      entity_name: suppliers
      io_schema:
        input_schema:
          type: object
          additionalProperties: false
          required: [supplier_id]
          properties:
            supplier_id: { type: string }
            reason: { type: string }
      effects:
        - entity: field
          mode: write
          impact_type: state_transition
          selectors: { dataset: suppliers, field_names: [status] }
          set_value: Blocked
  rules:
    - id: dtp/rule_blocked_supplier_guard
      title: 阻断供应商拦截
      severity: warn
      when: { entity: dataset }
      constraint: { type: security }
      message: 供应商状态为 Blocked 时，不应被推荐为可下单对象。
      remediation: 先解冻供应商或选择其他供应商。
```

### 2.3 建模原则：五个来自实践的约束

OSI 建模不是把数据库表直接映射过来，而是需要遵循以下原则：

1. **对象优先（Object-first）**：先把业务对象/事实粒度说清楚，再映射表。如果业务说"采购申请"，而不是"PR 表"，就先建 `purchase_requisitions` dataset。
2. **身份先行（Identity-first）**：先统一主键/唯一键与跨系统对齐，再谈指标口径。跨系统身份不对齐，relationship 再漂亮也会断。
3. **关系可计算（Relationship-as-Graph）**：BOM/订单链/事件链用 `relationships` 显式表达，确保可追溯，而不是藏在 join 条件里。
4. **指标下沉到可行动粒度**：指标口径写清楚，且能落到"能行动的粒度"。如果"按时交货率"没有定义"按时"的截止时间，它就不是一个可行动的 metric。
5. **闭环工作流**：监控→调查→分派→执行→回写→复盘，让模型不仅能"算"，还能"做"。

### 2.4 门禁校验：validate.py

生成的 YAML 必须通过 OSI 官方校验脚本：

```bash
python OSI/validation/validate.py semantic_model.yaml --summary
```

校验维度包括：Schema 结构与类型、dataset/field/metric/relationship 重名、relationship 引用有效性、SQL 表达式语法。**校验不通过，不交付最终版。**

## 三、端到端 pipeline：从业务域到语义本体的完整示例

以「需求到采购（DTP）」业务域为例，展示 pipeline 全流程：

### 3.1 Step 1：EA 业务拆解（输出分析结果）

**Root Drivers**：

- RD-1 供应可见：知道有没有、谁能供、多少钱
- RD-2 合规可控：预算合规、资质合规、审批合规
- RD-3 履约可追：订单到没到、货收没收到、收不到怎么办

**能力地图**（DTP 域）：

- 需求感知 → 寻源执行 → 采购执行 → 履约确认 → 绩效治理

**核心流程主干**：

```
需求提出 → 库存查重 → 供应商寻源 → 比价/定价 → 合同签订
→ 请购审批 → 订单下达 → 交期跟踪 → 收货验收 → 入库/结算
```

**Event Catalog**：

`PRCreated` → `PRApproved` → `POSent` → `POReleased` → `GRRequested` → `GRPosted` → `InvoiceMatched` → `PaymentApproved`

### 3.2 Step 2：OSI 模型生成（基于 EA 拆解产出）

**datasets（来自能力地图中的核心对象）**：

- `suppliers`（供应商主数据）
- `materials`（物料主数据）
- `purchase_requisitions`（请购单头表）
- `purchase_requisition_items`（请购单行项目）
- `purchase_orders`（采购订单）
- `goods_receipts`（收货单）
- `invoices`（发票）

**relationships（来自 Event Catalog 与流程链路）**：

- `pr_items → pr_header`（行 → 头）
- `pr → suppliers`（请购 → 供应商）
- `pr → materials`（请购 → 物料）
- `po → pr`（采购订单 → 请购单）
- `gr → po`（收货单 → 采购订单）
- `invoices → po`（发票 → 采购订单）

**metrics（来自 KPI 推导）**：

- `pr_count`：请购单数量
- `po_count`：采购订单数量
- `on_time_delivery_rate`：按时交货率
- `avg_procurement_lead_time`：平均采购交期

**behavior（来自治理控制点）**：

- `suppliers/block`：冻结供应商
- `suppliers/unblock`：解冻供应商
- `purchase_requisitions/approve`：审批请购单
- `purchase_orders/release`：释放采购订单

**rules（来自合规控制点）**：

- 阻断状态为 Blocked 的供应商参与采购
- 超过预算阈值的请购单必须上报审批
- 三单匹配（PO / GR / Invoice）不一致时不得过账

## 四、两条 Skill 的分工与边界

| | EA 业务拆解 | OSI 模型生成 |
|---|---|---|
| **定位** | 业务理解层：从需求到结构 | 技术建模层：从结构到可执行语义 |
| **输入** | 业务域描述 / 流程图 / 制度文档 | EA 拆解产出 / 数据源表结构 |
| **核心输出** | 《分析结果》+ 《逻辑推导》 | semantic_model YAML + validate.py 通过 |
| **回答的问题** | "这个业务域要做什么？有哪些能力？" | "这些能力用什么对象/关系/指标/动作表达？" |
| **不做什么** | 不生成代码、不映射表结构 | 不替代业务分析（依赖 EA 拆解在前） |
| **适用阶段** | 业务梳理期 / 需求冻结前 | 建模期 / 数据接入后 |

两者衔接的**关键接口**是 EA 拆解的"数据架构（DA）"层——它定义了主数据与交易数据对象，这些对象直接成为 OSI 的 datasets；EA 拆解的"Event Catalog"直接成为 OSI 的 relationships。

---

## 结语

EA 业务拆解解决的是"业务侧说清楚"的问题，OSI 模型生成解决的是"技术侧建标准"的问题。

两者叠加，让"从业务域到语义本体"这条链路第一次有了可复用的 pipeline——不需要每次都从零理解业务，也不需要每次都拍脑袋建模。

业务团队给出一个领域，EA 拆解把它变成结构化资产，OSI 模型把它变成可执行的语义本体。

这两条 Skill 合在一起，就是"业务 → 本体"的完整闭环。
