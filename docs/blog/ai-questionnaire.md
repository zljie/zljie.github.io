# AI 问卷信息采集 — 产品设计实践

Published: January 2024

## 项目背景

在房屋维修业务中，用户提交维修申请时需要提供详细信息，以便维修团队判断：

- 维修场景类型（漏水、墙面、电路等）
- 紧急程度
- 预估所需材料和工具
- 是否需要预约专业人员

传统的做法是提供一份静态表单，字段固定且繁多。用户往往：

1. 不清楚自己的问题属于哪种场景
2. 填写的信息不完整或不准确
3. 面对大量字段产生畏难情绪，导致表单放弃率高

## 问题分析

### 用户旅程中的摩擦点

通过用户访谈和数据分析，我们发现：

- **首次提交**：用户往往只有"水龙头漏水"这样的简单描述
- **信息缺口**：维修团队需要"漏水位置"、"水压情况"等详细信息
- **来回沟通**：平均每个工单需要 2-3 次电话沟通补充信息
- **无效上门**：约 15% 的上门维修因信息不足而白跑一趟

### 核心诉求

用户需要的是**渐进式引导**，而非一次性填写所有信息。系统需要：

1. 从用户的简单描述中理解场景
2. 根据已知信息，智能推断还需要什么
3. 以最少的交互次数，获取足够的信息

## AI 问卷方案设计

### 核心思路：问卷即对话

传统问卷是**静态的**，AI 问卷是**动态的**。

```
传统问卷：
  问题1 → 问题2 → 问题3 → ... → 提交
           ↓(跳题)
        问题4 → 问题5 → ...

AI 问卷：
  初始问题 → AI分析 → 智能追问 → AI分析 → 智能追问 → ... → 提交
```

### 多端式采集策略

根据不同场景，设计了不同的采集模式：

| 场景 | 初始问题数 | AI 追问策略 |
|------|-----------|------------|
| 紧急报修 | 1-2 个 | 直接指派工单，后续补全 |
| 常规维修 | 3-5 个 | 引导式采集，深度了解 |
| 定期维护 | 5-7 个 | 全面信息收集，预防性建议 |

### 智能追问引擎

核心是结合**系统知识库**和**已收集信息**的推理引擎：

```
用户输入 → 意图识别 → 场景分类 → 知识匹配 → 生成追问
    ↓
意图分类：['漏水', '墙面', '电路', '门窗', '其他']
    ↓
知识库检索：漏水场景需要了解 "位置、水量、持续时间、影响范围"
    ↓
追问生成：根据已有信息，生成2-3个最相关问题
```

### 知识库构建

为支撑智能追问，构建了两层知识库：

#### 场景知识库

每个维修场景的标准化信息模板：

```json
{
  "scenario": "漏水维修",
  "required_fields": ["漏水位置", "水量大小", "持续时间"],
  "optional_fields": ["水压情况", "邻居影响", "历史维修记录"],
  "questions": [
    "请问漏水的大概位置是哪里？",
    "水量大吗？是一直在漏还是间歇性的？",
    "这种情况持续多长时间了？"
  ]
}
```

#### 追问规则库

场景与追问的映射规则，支持条件组合：

```json
{
  "conditions": [
    "scenario = '漏水'",
    "location CONTAINS '天花板'",
    "water_amount > '中等'"
  ],
  "questions": [
    "天花板漏水通常和楼上住户有关，请问您楼上的房屋目前有人住吗？"
  ]
}
```

## 技术实现

### AI 服务层

采用大语言模型（LLM）作为追问生成的推理引擎：

1. **Prompt 工程**：将场景知识库注入 prompt，引导模型生成符合规范的追问
2. **输出校验**：对模型的输出进行格式校验和敏感词过滤
3. **降级策略**：模型不可用时，回退到基于规则的知识库匹配

### 性能优化

- 追问生成控制在 **2 秒内**完成，保证用户体验
- 知识库预加载到内存，减少检索延迟
- 异步处理非关键路径（用户感知不到）

## 成果数据

| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| 有效问卷收集率 | 35% | 70% | +100% |
| 现场勘探次数（每10单） | 5.2 次 | 3.0 次 | -42% |
| 平均完单时间 | 4.2 天 | 3.1 天 | -26% |
| 用户满意度 | 3.2/5 | 4.5/5 | +41% |

## 总结

AI 问卷的核心价值在于**将单向的信息收集转变为双向的对话交互**。用户不再需要面对一堆陌生的字段，而是像和一个专业人士沟通一样，逐步明确问题、获取帮助。这既提升了信息采集的质量，也改善了用户体验。
