# 公共服务组敏捷转型 — 敏捷教练实践记录

Published: October 2020 - March 2021

## 项目背景

因企业战略调整，原有的公共产研、数据治理、数据工具三个部门合并为**公共服务组**，统一为各业务线提供底层技术服务。

合并后面临的核心问题：

- **流程不统一**：三个团队各自为政，工作流程差异大
- **技术栈陈旧**：部分团队仍使用 .Net 技术，与云原生趋势脱节
- **交付效率低**：从需求到上线周期长，业务部门满意度不高
- **知识断层**：关键岗位人员离职风险高

我作为敏捷教练，负责推动这次转型。

## 转型目标

| 维度 | 当前状态 | 目标状态 |
|------|----------|----------|
| 开发模式 | 传统瀑布 | 敏捷迭代 |
| 技术栈 | .Net | Java + 云原生 |
| 交付周期 | 4-8 周/需求 | 2 周/Sprint |
| 团队协作 | 部门隔离 | 跨职能协作 |

## 转型路径

### 第一阶段：调研与诊断

首先对三个团队进行了全面调研：

- **流程审计**：梳理现有的需求、开发、测试、上线流程
- **痛点访谈**：与 30+ 位团队成员进行一对一访谈
- **数据收集**：统计历史项目的交付周期、缺陷率、客户满意度

关键发现：

1. 需求变更频繁，但变更管理流程缺失
2. 测试环节严重滞后，是交付的主要瓶颈
3. 团队间存在"部门墙"，协作效率低
4. 技术债务积累，系统稳定性差

### 第二阶段：敏捷工作坊

组织了三场全员参与的敏捷工作坊：

#### 工作坊 1：愿景与目标对齐

> "我们这三个月要达成什么？"

通过 group discussion，团队共同定义了 Sprint 的 MVP 目标和成功标准。

#### 工作坊 2：流程设计

> "我们怎么做这件事？"

基于 Scrum 框架，结合公共服务组的实际情况，设计了简化的敏捷流程：

```
Product Backlog → Sprint Planning → Sprint (2 weeks)
                    ↓
              Daily Standup
                    ↓
              Sprint Review → Sprint Retrospective
                    ↓
              Product Backlog Refinement
```

#### 工作坊 3：角色定义

> "谁负责什么？"

明确各角色职责：

- **Product Owner**：对接业务需求，负责优先级排序
- **Scrum Master**：协调团队，解决障碍
- **Team Member**：自组织完成任务

### 第三阶段：技术转型

#### .Net → Java

根据业务需求和团队技能，制定了渐进式迁移方案：

1. **并行阶段**（1-3 月）：新功能使用 Java 开发，老功能继续维护
2. **替换阶段**（4-6 月）：逐步替换核心模块
3. **收尾阶段**（7-8 月）：完成 .Net 代码库下线

技术选型：

- 框架：Spring Boot
- 数据库：MySQL + Redis
- 消息队列：Kafka
- 容器化：Docker + Kubernetes

### 第四阶段：持续改进

引入**数据驱动**的改进机制：

- Sprint 燃尽图：跟踪进度，及时发现风险
- 发布后检查表（PIR）：每次上线后复盘
- 团队健康度检查：每季度评估团队状态

## 转型成果

### 团队层面

| 指标 | 转型前 | 转型后 | 改善 |
|------|--------|--------|------|
| 敏捷团队数量 | 0 | 3 | - |
| Sprint 准时交付率 | 45% | 85% | +40pp |
| 代码评审覆盖率 | 20% | 95% | +75pp |
| 技术债务修复速度 | 慢 | 持续 | 显著改善 |

### 业务层面

#### 数据分析组

- **财报生产周期**：从 7 天缩短至 5 天
- **数据检验自动化**：新增 12 项自动化校验规则
- **人工干预减少**：重复性校验工作减少 60%

#### 估价组

- **估价周期**：从 5 天缩短至 3 天
- **团队效率**：1 人可完成原来 2 人的工作量
- **客户满意度**：提升 35%，成功替代客户原有团队

## 经验总结

### 敏捷落地的关键

1. **领导层支持**：敏捷转型需要自上而下的支持，否则阻力重重
2. **因地制宜**：不要照搬框架，要根据团队实际情况调整
3. **持续教育**：敏捷是思维方式的转变，需要持续培训和实践
4. **数据验证**：用数据说话，让成果可见

### 常见误区

- **敏捷 = 不用文档**：敏捷强调 Working Software，但必要的文档仍需保留
- **敏捷 = 不要计划**：敏捷拥抱变化，但不是没有计划
- **一蹴而就**：敏捷转型是一个持续的过程，不要期望一次工作坊就能完成
