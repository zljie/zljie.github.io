## 🎯 核心主题：SSE 流式响应：后端跑通了，前端一个字没显示——四次调试的血泪史

### 🟢 S (Situation) - 稳定的前度平衡

> *用 2-3 句话交代原本平稳、大家都习以为常的现状。*

凌晨两点的办公室，键盘声稀稀落落。AgentScope 的 `reply_stream` 跑了三个月，测试脚本里 print 出来的东西整整齐齐，"ThinkingBlock"、"TextBlock"、token 数量，一个不缺。本地 `python app.py` 起来，curl 往 `/chat` 一打，event stream 哗哗往外冒，`data: {"content": "嗯"}` 排成一列。后端稳了。Railway 部署流水线绿了，Docker 镜像推上去了。去前端看一眼——deep-chat 的气泡里空空如也，发送按钮灰着，控制台干干净净，连一个 warning 都没有。

"大概是网络问题吧。"

### 🔴 C (Complication) - 平衡的剧烈打破

> *用 2-3 句话撕开痛苦的口子，描述突然发生的变量或致命痛点。*

不是网络问题。

四次调试，四次心态崩盘，每一次的原因都跟「API 文档没写」或者「组件默认行为藏得太深」有关：

**第一坑：AgentScope v1.x → v2.0，流式 API 事件类型集体叛变**

三个月前写 `agent.py` 时，用的还是 v1.x，`ThinkingBlock` 和 `TextBlock` 对象从 `reply_stream` 里直接蹦出来，类型判断一行行写得工整。升级 AgentScope 到 v2.0 后重新跑，`unknown event type` 铺满日志——所有事件都不认了，delta 流全进黑洞，AI 回答一个字符都 yield 不出来。

**第二坑：Starlette Jinja2Templates 在 Docker 里字典缓存爆炸**

本地 dev 环境一切正常，`python app.py` 起来就能看到 admin 页面。镜像推到 Railway，第一笔请求进来，500。日志里一行 `TypeError: unhashable type: 'dict'`——Starlette 的 Jinja2Templates 在 cache key 里塞了 dict，直接短路。本地永远复现不了，因为本地是直接 `python app.py` 跑，Docker 里是从 `/app` 目录起的，路径全乱了。

**第三坑：deep-chat 内置 SSE 解析器只认 `message`，自定义事件名被静默吞掉**

后端 SSE 格式完全正确，`event: think` / `event: content` / `event: done` 写得标准，`data` 字段用 `{"content": "..."}` 格式。但 deep-chat 的内置 SSE 处理器只监听 `event: message` 和默认无事件名行，期望的数据格式是 `{"text": "..."}`。我们的每一行 `event: content` + `data: {"content": "..."}` 到了 deep-chat 那边，全被静默丢弃——没有 error，没有 warning，API 返回 200，数据就躺在网络层里，前端根本看不见。

**第四坑：uvicorn 异步循环与部署环境不兼容，SSE 流偶尔断流**

修完前三坑，流式输出开始工作了，但偶尔莫名其妙中断。查了一圈，问题出在 uvicorn 的异步循环和 HTTP 实现上——某些 Railway 运行时环境里默认配置和容器网络栈打架。

**破局一：AgentScope 2.0 的 Delta 事件架构**

v2.0 的 `reply_stream` 在流式输出过程中 yield 的是增量事件，而非完整的 Block 对象。必须从 `DeltaEvent` 逐 token 累积到缓冲区，实时 yield 到 SSE：

```python
from agentscope.event import (
    ThinkingBlockDeltaEvent,
    TextBlockDeltaEvent,
    ThinkingBlockEndEvent,
    TextBlockEndEvent,
)

think_buf: list[str] = []
text_buf: list[str] = []
think_yielded = False
content_yielded = False

async for event in self.agent.reply_stream(msg):
    if isinstance(event, ThinkingBlockDeltaEvent):
        think_buf.append(event.delta)
        yield think(event.delta)        # 实时 yield 增量片段
        think_yielded = True
    elif isinstance(event, TextBlockDeltaEvent):
        text_buf.append(event.delta)
        yield content(event.delta)      # 实时 yield 增量片段
        content_yielded = True
    elif isinstance(event, ThinkingBlockEndEvent):
        print(f"[LLM STREAM] ThinkingBlockEnd, think_buf len={len(''.join(think_buf))}")
    elif isinstance(event, TextBlockEndEvent):
        print(f"[LLM STREAM] TextBlockEnd, text_buf len={len(''.join(text_buf))}")

# Fallback：某些模型不触发 delta 流，从最终 context 中提取完整 Block
if not think_yielded and not content_yielded and self.agent.state.context:
    last_msg = self.agent.state.context[-1]
    for block in last_msg.content:
        if hasattr(block, "thinking"):
            yield think(block.thinking)
        elif hasattr(block, "text"):
            yield content(block.text)

if think_yielded:
    yield think_done()
```

`StartEvent` / `EndEvent` 只做边界日志，不 yield 内容；`Block` 对象（`ThinkingBlock` / `TextBlock`）在 v2.0 中仅存在于最终消息中，不在流式过程中出现。

**破局二：原生 Jinja2 替换 Starlette Jinja2Templates，路径改用绝对路径**

```python
import jinja2

# 始终相对于项目根目录解析，不依赖 cwd
_templates_dir = Path(__file__).parent.parent / "templates"
_jinja_env = jinja2.Environment(
    loader=jinja2.FileSystemLoader(str(_templates_dir)),
    autoescape=jinja2.select_autoescape(["html", "xml"]),
)

# 在 endpoint 中：
template = _jinja_env.get_template("admin/base.html")
return HTMLResponse(template.render(request=request))
```

使用 `Path(__file__).parent.parent / "templates"` 拿到项目根目录的绝对路径，Docker 容器无论从哪个目录启动都能找到。

**破局三：deep-chat `connect.handler` 完全接管 SSE 解析**

deep-chat 的内置 SSE 处理器无法识别自定义事件名，绕开它的唯一方式是使用 `handler` API 手动消费 `ReadableStream`：

```javascript
chatElement.connect = {
    url: '/chat',
    method: 'POST',
    headers: {'Content-Type': 'application/json'},
    handler: function(body, signals) {
        fetch('/chat', {
            method: 'POST',
            headers: {'Content-Type': 'application/json'},
            body: JSON.stringify(body),
        }).then(function(response) {
            if (!response.ok) {
                signals.onResponse({error: 'Request failed'});
                return;
            }
            signals.onOpen();

            // 手动消费 ReadableStream，逐行解析 SSE
            const es = response.body.getReader();
            const decoder = new TextDecoder();
            let buffer = '';

            function processLine(line) {
                if (line.startsWith('event: ')) {
                    buffer = '__event__' + line.slice(7);
                } else if (line.startsWith('data: ')) {
                    const data = line.slice(6);
                    const eventType = buffer.startsWith('__event__')
                        ? buffer.slice(9)
                        : 'default';

                    if (eventType === 'think' || eventType === 'think_done') {
                        // 跳过 thinking 事件
                    } else if (eventType === 'content') {
                        try {
                            const json = JSON.parse(data);
                            const text = json.content || '';
                            if (text) signals.onResponse({text: text});
                        } catch(e) {}
                    } else if (eventType === 'done') {
                        signals.onClose();
                    }
                    buffer = '';
                }
            }

            function read() {
                es.read().then(function(result) {
                    if (result.done) { signals.onClose(); return; }
                    buffer += decoder.decode(result.value, {stream: true});
                    const lines = buffer.split('\n');
                    buffer = lines.pop() || '';
                    lines.forEach(processLine);
                    read();
                }).catch(function(err) {
                    signals.onResponse({error: err.message});
                    signals.onClose();
                });
            }
            read();
        });
    },
};
```

关键点：SSE 行可能跨多个 TCP 包到达，必须 buffer 后按 `\n` 分割；`event:` 行和 `data:` 行的跨行状态需要用 buffer 变量维护；`signals.onOpen()` / `signals.onResponse({text: ...})` / `signals.onClose()` 是 deep-chat handler API 的标准信号。

**最终 SSE 事件规范（与前端 handler 对齐）：**

```
event: think
data: {"content": "用户问的是行程计算..."}

event: think_done
data: {"status": "done"}

event: content
data: {"content": "根据你的描述，..."}

event: done
data: [DONE]
```
