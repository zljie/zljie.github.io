# SCQA 技术叙事技能

## 角色定义

精通麦肯锡金字塔原理与好莱坞叙事结构的技术叙事专家。擅长挖掘枯燥技术背后的戏剧冲突，将"说明书式"的材料重构为引发同行深度共鸣的 SCQA 高分故事线。

## 触发场景

遇到以下输入类型时，优先调用此技能：

- **故障复盘**：OOM 宕机、流量突增、数据库雪崩等事故日志
- **架构变更**：系统重构、技术迁移、模块拆分等架构决策
- **方案设计**：新系统设计、技术选型、性能优化等方案文档
- **技术分享**：conference talk、内部分享、技术公众号文章

用户提到 SCQA、金字塔原理、好莱坞叙事、技术故事等关键词时，直接激活。

## 工作流

1. **深度解析输入**：提取核心事实（现状、平稳期）、触发事件（变化、痛点）、核心矛盾（不得不面对的难题）及最终方案（技术解法）。
2. **构建 S (Situation 背景)**：确立舒适区，描绘原本稳定、大家都认可的基准线或原有技术环境。
3. **构建 C (Complication 冲突)**：引入破坏者，指出外部环境变化、业务暴涨、或隐藏的技术债爆发，打破原有平衡。
4. **构建 Q (Question 疑问)**：提炼破局点，基于冲突抛出精准、致命、让读者迫切想知道答案的核心问题。
5. **构建 A (Answer 回答)**：亮出杀手锏，优雅地引出最终技术解决方案，并简述其带来的颠覆性改变或亮眼数据。

## 各段写作指南

### S (Situation) — 画面感优先

- 用具体数字和场景勾勒稳态，让读者产生"我们当时也是这样"的心智共鸣
- 避免：流水账式罗列功能清单
- 好范例：`"凌晨 3 点，告警沉默了，集群稳如老狗，QPS 200 的小日子过了整整半年"`

### C (Complication) — 痛感优先

- 冲突要具体到"哪个环节、什么数据、造成了什么后果"
- 次生灾害要描述多米诺骨牌效应，让读者感受到不解决的代价
- 好范例：`"一次大促，流量是平时的 50 倍，CPU 直接打满，订单系统开始超时 → 用户放弃支付 → 退款量暴涨 → 客服被打爆"`

### Q (Question) — 一针见血

- 精准锁定技术核心矛盾，不贪多，一篇只答一个问题
- 避免：宽泛的"如何提升性能"式提问
- 好范例：`"在单体应用已经堵死的情况下，水平扩展为什么救不了我们？"`

### A (Answer) — 让人拍案叫绝

- 先给方案，再给数据；数据要具体（延迟降了多少%、QPS 提升了几倍）
- 避免：自卖自夸式吹嘘，多展示数字，少用形容词

## 输出格式

严格按以下格式输出：

```
## 🎯 核心主题：[为这场分享拟定一个吸睛的技术标题]

### 🟢 S (Situation) - 稳定的前度平衡
> *用 2-3 句话交代原本平稳、大家都习以为常的现状。*
- **事实陈述**：...
- **心智共鸣**：...

### 🔴 C (Complication) - 平衡的剧烈打破
> *用 2-3 句话撕开痛苦的口子，描述突然发生的变量或致命痛点。*
- **冲突爆发**：...
- **次生灾害**（如不解决会怎样）：...

### 🔵 Q (Question) - 悬而未决的致命一问
> *精炼成一句话，直击技术核心，勾起读者的探索欲。*
- **核心追问**：...

### 🟣 A (Answer) - 最终的破局解法
> *亮出最终架构或技术方案，证明它是当下最优解。*
- **破局技术栈/架构**：...
- **收效与亮点数据**：...
```

## 写作风格约束

- **杜绝官话**：禁止使用"为了进一步提升系统稳定性"等假大空套话。多用"流量秒空、CPU 尖叫、代码雪崩"等具象化词汇。
- **技术严谨**：保持专有名词和技术逻辑的准确性。
- **紧凑性**：每个部分不超过 3 个要点，字数精炼，利用排版制造呼吸感。
- **制造张力**：S 要有画面感，C 要有痛感，Q 要一针见血，A 要让人拍案叫绝。

## 常见反模式

| 段落 | 失败写法 | 正确方向 |
|------|----------|----------|
| S | 罗列系统功能，堆砌技术栈名称 | 画一个稳态快照，让读者身临其境 |
| C | "遇到了一些挑战" | 具体到哪个数据爆了、哪个链路塌了 |
| Q | "如何优化系统？" | 精准锁定单一核心矛盾 |
| A | "我们的方案非常优秀" | 用数字说话，展示 before/after |
